\section{Implementation}
\label{sec:implementation}

This section walks the implementation top-to-bottom, from the tree-sitter

verification hookup, the three-tier coverage harness, the editor surfaces,
and the fidelity-diagnostic pass. Figure~\ref{fig:architecture} sketches
the data flow; engineering counts per layer appear in

\subsection{Parser and AST}
\label{subsec:parser}
The grammar is implemented in tree-sitter~\cite{treesitter}, which gives
us incremental parsing, error recovery, and a uniform interface for the
language server, the MCP server, and the CLI to consume the same parse
tree. The grammar source is a single \texttt{grammar.js} file ($\sim$900
SLOC). Tree-sitter compiles it into a C parser plus Python bindings; the
runtime overhead of parsing a typical statute file (200 SLOC of Yuho) is
sub-millisecond.

A Python AST builder (\texttt{src/yuho/ast/builder.py}) lifts the
concrete syntax tree into immutable frozen-dataclass nodes
(\texttt{src/yuho/ast/nodes.py}, $\sim$1.4k SLOC for nodes alone). Every
node carries a \texttt{source\_location} field (file, line, column,
byte-range) for diagnostic reporting, and every node implements
\texttt{accept(visitor)} for double-dispatch over the analyser
hierarchy. Frozen dataclasses are deliberate: every analyser, transpiler,
diagnostic pass, and verification target consumes the same AST, so
structural sharing and equality-by-value are first-order requirements
rather than optimisations. The full AST package is approximately 7.4k
SLOC across nodes, builder, visitors, and structural validators.

\subsection{Six transpilers}
\label{subsec:transpilers}
Each transpiler is a Visitor over the AST. The six current targets cover
distinct downstream use cases:

\begin{description}
\item[JSON.] Structural dump matching the AST. Used by tooling consumers
that prefer parsing JSON to depending on the Python AST classes.
Round-trips via a published JSON Schema.
\item[Controlled English.] Template-driven natural-language rendering.
Outputs prose that a non-technical reader can audit against the canonical
statute text without learning Yuho syntax.
\item[\LaTeX{}.] Camera-ready statute renders with footnoted SSO anchors,
typeset penalties, and structurally-numbered illustrations. Compiles
straight to PDF via \texttt{pdflatex}.
\item[Mermaid.] Decision-tree flowcharts for elements and exceptions ---
useful for teaching and for the documentation site.
\item[Alloy.] A relational model suitable for bounded model-checking.
The Alloy transpiler is the largest of the six because it carries the
type-mapping logic from Yuho elements to Alloy signatures and facts;
its share of the 4{,}200-SLOC transpiler aggregate in
Table~\ref{tab:sloc} is roughly 60\%.
\item[DOCX.] Office Open XML output, written as a stdlib-only minimal
WordprocessingML package (\texttt{zipfile} + a fixed set of XML parts;
no \texttt{python-docx} dependency). The DOCX transpiler is the
plumbing for the in-progress Microsoft Word add-in.
\end{description}

The transpilers share a registry pattern (\texttt{src/yuho/transpile/registry.py})
that lazy-loads a target on first request and caches the instance. This
matters in the LSP and MCP servers, where multiple transpile requests
land in the same process.

\subsection{Verification: Z3 and Alloy}
\label{subsec:verify}
The verification stack checks well-formedness of the \emph{encoding}, not
legal correctness; properties about ``the law'' are out of scope and are
discussed in \S\ref{sec:limitations}. Two solvers, two purposes.
Z3~\cite{demoura2008z3} powers the
exception-priority validator: given an encoded section with a list of
prioritised exceptions, the solver checks that the priority ordering is
well-formed (no cycles, no priority-inversion under composition with
Chapter IV general exceptions) and that the rewriting semantics is
deterministic for every reachable element configuration. The Z3 hookup
also drives a cross-section conflict detector: given two sections that
plausibly overlap (e.g.\ s299 and s300), the solver searches for an
element-binding that satisfies both, and reports the configuration if
one exists.

Alloy~\cite{jackson2002alloy} provides bounded model-checking. The Alloy
transpile target emits a signature for every element, fact for every
constraint, and predicate for every penalty branch. The Analyzer
(invoked via the \texttt{yuho verify --engine alloy} CLI subcommand)
runs bounded enumeration over the resulting model: the typical use is
to ask, ``does there exist a binding of size $\leq k$ that satisfies
the offence elements but violates the burden qualifier on the mens-rea
element?'' and inspect the counterexamples.

The verification stack is approximately 3.3k SLOC, split between the
Z3 solver wrapper, the Alloy emitter, and a handful of property
templates that encode the most common queries (consistency,
exception-priority well-formedness, mutual-exclusion of penalty
branches, and inter-section overlap).

\paragraph{Encoding the exception semantics correctly.} An early
implementation of the Z3 generator asserted, per encoded section,
the pair
$\mathit{conviction} \Leftrightarrow \mathit{all\_elements}$ and
$\mathit{exception\_fires} \Rightarrow \neg\mathit{conviction}$.
The pair is consistent in isolation but jointly entails
$\mathit{exception\_fires} \Rightarrow \neg\mathit{all\_elements}$,
which is doctrinally the wrong reading: the whole purpose of an
exception is to suppress conviction \emph{despite} the elements
being satisfied (a defendant raising a Chapter IV general defence
is not denying the elements; she is asking the court to excuse the
conduct). Under the original encoding, the satisfiability query
``does there exist a fact pattern in which the elements are
satisfied and this exception fires?'' returned UNSAT for
\statRawSections{}-equivalent sections that have any encoded
exception, making the counter-example explorer's reachability
diagnostic uniformly negative. The corrected encoding splits the
biconditional via a synthetic
$\mathit{elements\_satisfied}$ aggregate and asserts
$\mathit{conviction} \Leftrightarrow \mathit{elements\_satisfied}
\wedge \neg(\bigvee_i \mathit{fires}_i)$,
which preserves the original semantics when no exception fires and
admits the satisfied-but-excused case otherwise. The change moved
exception reachability across the corpus from $4/21$ sections with
at least one reachable exception to $21/21$ --- a confirmation that
the correction matches the doctrine and a reminder that the design
of structural verifications, even on small grammars, can sit
arbitrarily close to a logical inversion of the intended reading.

\subsection{Three-tier coverage harness}
\label{subsec:coverage}
Coverage is graded on a three-tier scale:
\begin{description}
\item[L1 (parse).] The file passes the tree-sitter parser without error,
and the AST builder produces a non-trivial \texttt{ModuleNode}.
\item[L2 (build + lint).] The AST passes static semantic checks: type
inference resolves cleanly, identifiers bind to declarations,
illustration labels are unique, and the fidelity-diagnostic pass
(\S\ref{subsec:fidelity}) emits no warnings.
\item[L3 (human-stamped).] A human (or a sufficiently capable agent
acting as a structured reviewer) has audited the encoding against the
canonical SSO text using an 11-point checklist, and the audit has
stamped \texttt{[verification].last\_verified} in the per-section
\texttt{metadata.toml}.
\end{description}

The L1+L2 tiers are computed mechanically by
\texttt{scripts/coverage\_report.py}, which emits a JSON dashboard
(\texttt{library/penal\_code/\_coverage/coverage.json}) and a Markdown
summary. The L3 tier is dispatcher-driven
(\texttt{scripts/phase\_d\_l3\_review.py}), which renders the 11-point
prompt template against per-section context and invokes \texttt{codex
exec} agents in parallel; each agent either stamps the section or
appends a flag to \texttt{library/penal\_code/\_L3\_flags.md} for human
review. The dispatcher supports resumable progress logs so a long run
can be paused and continued without re-doing stamped sections.

\subsection{Editor surfaces}
\label{subsec:editors}
Three editor-facing surfaces share the same AST and analyser stack.

\paragraph{Language Server.} A pygls-based Language Server
(\texttt{src/yuho/lsp/}, $\sim$3.2k SLOC) provides hover, inlay hints,
context-aware completion, code lenses, diagnostics, document symbols,
and basic refactorings (rename within file). Hover on a \texttt{statute N}
header surfaces the canonical SSO URL, the marginal note, a $\sim$280-character
canonical excerpt from the scrape, and L1/L2/L3 coverage badges.
Completion narrows to context-appropriate keywords: after \texttt{penalty},
the candidate set is \texttt{cumulative} / \texttt{alternative} /
\texttt{or\_both} / \texttt{when}; after \texttt{fine :=}, the candidate
is \texttt{unlimited}; after \texttt{caning :=}, the candidate is
\texttt{unspecified}. Inlay hints summarise structural counts per statute
(``$\langle 3$ elements $\cdot$ 16 illustrations $\cdot$ 2 subsections
$\cdot$ effective 1872-01-01$\rangle$''). Code lenses surface L1/L2/L3
status badges, ``Run tests'', and ``Transpile to English'' actions.

\paragraph{Model Context Protocol server.} A FastMCP-based server
(\texttt{src/yuho/mcp/}, $\sim$2.7k SLOC) exposes the same workflow as
machine-callable \emph{tools}, \emph{resources}, and \emph{prompts}
suitable for AI clients (Claude Desktop, Claude Code, Codex CLI,
Cursor). Tools include \texttt{yuho\_check}, \texttt{yuho\_transpile},
\texttt{yuho\_section\_pair} (returns the canonical text alongside the
encoded AST for side-by-side reads), \texttt{yuho\_run\_l3\_review}
(runs the same 11-point checklist used by the L3 dispatcher), and
\texttt{yuho\_apply\_flag\_fix} (rewrites a flagged section using the
new grammar primitives). Resources include the library index, raw
canonical text per section, and the current grammar summary. Prompts
include \texttt{find\_fidelity\_issues}, \texttt{recommend\_l3}, and
\texttt{encode\_new\_section}.

\paragraph{VS Code extension.} An extension under
\texttt{editors/vscode-yuho/} (TypeScript, $\sim$200 SLOC for
\texttt{extension.ts} plus snippets, syntax highlighting, and
configuration) wraps the Language Server, registers commands
(``Open SSO page'', ``Run yuho check'', ``Transpile to Mermaid'',
``Show coverage dashboard''), and renders a status-bar coverage indicator
that updates as \texttt{coverage.json} changes on disk.

\subsection{Fidelity diagnostics}
\label{subsec:fidelity}
Four diagnostic passes cross-check the encoded AST against the canonical
scrape (\texttt{library/penal\_code/\_raw/act.json}), catching the
fidelity errors that mass-encoding most frequently produced:

\begin{itemize}
\item \textbf{Illustration-count mismatch (\textsf{G4}).} Warn if the
encoded section has strictly fewer illustrations than the canonical
statute defines. The check is conservative: more illustrations is
acceptable (subsection-level worked examples are sometimes flattened
into the parent block); fewer is a sign of an incomplete encoding.
\item \textbf{Fabricated fine cap.} Warn if the canonical statute text
matches the pattern ``with fine'' without a numeric value, but the
encoding uses a numeric \texttt{fine := \$X .. \$Y} range. The correct
encoding in this case is the \textsf{G8} sentinel \texttt{fine :=
unlimited}; a numeric range is a fabrication.
\item \textbf{Fabricated caning range.} Warn analogously when the
canonical text says ``shall be liable to caning'' with no stroke count,
but the encoding uses a numeric \texttt{caning := N .. M strokes} range.
The correct encoding is \texttt{caning := unspecified} (the \textsf{G14}
sentinel).
\item \textbf{Disjunctive-connective mismatch (\textsf{G11}).} Warn if
the encoding wraps elements in \texttt{all\_of} but the canonical text
uses ``or'' between the corresponding elements, or vice versa. The check
is heuristic --- it pattern-matches connectives near the element markers
in the canonical text --- but it catches the most common conjunction-vs-
disjunction encoding error.
\end{itemize}

The diagnostics emit at L2: a section that triggers any of them does
not pass L2, and the coverage dashboard reflects the failure. Each
diagnostic is accompanied by a \emph{did-you-mean}-style suggestion ---
\texttt{fine := \$0 .. \$50000} \,$\rightarrow$\, \texttt{fine := unlimited}, for
example --- and code-action support in the LSP autofixes the issue with
one keystroke.

\subsection{Did-you-mean for unknown directives}
A separate diagnostic uses Python's \texttt{difflib} to suggest the
nearest known keyword when the parser rejects an unknown identifier.
Encoders typing \texttt{actu\_reus} (a typo) or \texttt{element\_groups}
(plural) get an inline ``did you mean \texttt{actus\_reus}?'' /
``\texttt{element\_group}?'' suggestion. The keyword set is the union
of every reserved word in the grammar, every element kind, every
penalty combinator, every deontic operator, every exception field,
and every annotation name --- approximately fifty terms.

\subsection{Analytical tools over the encoded corpus}
\label{subsec:analytical_tools}

Two analytical surfaces are layered on top of the verification stack
and the slim corpus. Both deliberately re-use existing core machinery
rather than introducing new translation layers, and both are exposed
via the CLI and the MCP server.

\paragraph{Counter-example explorer.} The explorer
(\texttt{src/yuho/explore/}) takes an encoded section and surfaces the
fact-pattern \emph{witnesses} the Z3 generator can construct over its
element / exception structure. Concretely, four queries are run
against the per-section conviction Bool that the verifier already
emits (\S\ref{subsec:verify}):
\emph{satisfying} scenarios (assignments to element-satisfaction Bools
that make conviction reachable, blocked-and-re-checked to enumerate
distinct witnesses up to a bound);
\emph{borderline} scenarios (for each leaf element, a model where every
other element is satisfied but this one is negated --- a witness that
the element is load-bearing in the encoded sense);
\emph{exception coverage} (for each encoded exception, a check that
some element-satisfying assignment also fires the exception --- proving
the exception is reachable rather than dead);
and the complement \emph{dead-exception} list (exceptions whose firing
is unsatisfiable under any element-satisfying assignment, suggesting an
overconstrained guard or unused branch).
The explorer adds no new constraint translation; the existing Z3 model
generator (\texttt{Z3Generator}) is its sole dependency. On s415 the
explorer produces, in $<200\,$ms, two distinct satisfying assignments
witnessing the \texttt{any\_of} mens-rea group (one with
\texttt{fraudulent}, one with \texttt{dishonest}), confirms all five
leaf elements are load-bearing, and reports zero exceptions to test.
The pedagogical use case is direct: a teacher can ask the tool to
exhibit the precise minimal pairs that distinguish a section's
elements, and a legal-tech builder can use the same enumeration to
seed automated test fixtures.

\paragraph{Charge recommender.} The recommender
(\texttt{src/yuho/recommend/}) takes a fact pattern (the same
\texttt{simulator/} schema, but with the \texttt{section} field
intentionally omitted) and ranks candidate Penal Code sections by
structural fit. It is a thin orchestrator: a cheap token-overlap
prefilter selects up to sixty candidate sections from the slim
corpus; for each, the existing fact-pattern simulator's
\texttt{evaluate} function is run to produce a per-element trace
(satisfied / suggested / unresolved / contradicted); the candidates
are scored by element coverage, with contradicted elements applying
a hard penalty; the top-$k$ are returned with the simulator's full
trace and an SSO link attached.
The recommender does not call any external language model and does
not access the network. Every result envelope carries an
\texttt{not\_legal\_advice: true} flag and a fixed disclaimer string
(\texttt{LEGAL\_DISCLAIMER}) that callers --- the CLI, the MCP tool,
or any downstream consumer --- are required to surface verbatim.
Validated against the simulator fixtures (encoded as a
regression-tested benchmark in
\texttt{tests/test\_recommender\_benchmark.py}), the recommender
ranks the canonical section first for the two positive single-section
fixtures (s415 classic-cheating, s378 theft); for a deliberate
negative fixture (s415 with-consent, where the supplied facts
contradict the deception element) the recommender correctly does
\emph{not} surface s415 as the top candidate; for a multi-offence-
family fixture (s319 hurt) the recommender places one of the bodily-
harm offences in the top three. A multi-section abetment fixture
(s107 / s109 / s120A) initially failed because element descriptions
in s107 use \texttt{instigation} / \texttt{intentional aid} where
fact patterns naturally write ``A abets B'' or ``A procures B'';
three improvements compose to close that gap: (a)~a one-hop walk of
the G10 reference graph, used to expand keyword-overlap candidates
with their incoming and outgoing references; (b)~a rule-based
stemmer that collapses tense and derivation differences
(\texttt{deceives}~/~\texttt{deception}~/~\texttt{deceit} all reduce
to a common root); and (c)~a small doctrinal synonym table covering
the abetment / aid / theft / deception clusters that the encoded
elements draw from. With all three the recommender surfaces the
abetment family in the top five for that fixture --- not s107
itself, which is purely definitional, but the actionable abetment
sections (s108 / s108A / s108B / s115 / s120A / s306) that the user
of a charge recommender would want to inspect first.

The recommender's coverage metric is structural by construction: it
measures element-overlap between facts and encoded grammar, not the
likelihood of conviction. The disclaimer is therefore not a polite
hedge but a load-bearing contract; \S\ref{sec:limitations} discusses
what the recommender is and is not.

calibrates a ``productive'' DSL-to-corpus ratio. The numbers are
included for transparency about effort and surface area, not as a
density argument.
